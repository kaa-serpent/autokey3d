## Commands

All commands must be run in the venv.

```bash
# Launch the Tkinter UI
python autokey_ui.py

# Launch the web app (http://localhost:5000)
python web_app.py

# CLI key generation (backward-compatible)
python AutoKey.py --bumpkey --profile profiles/AB-AB95.svg --definition definitions/AB-E20.scad
python AutoKey.py --blank   --profile profiles/AB-AB1.svg  --definition definitions/AB-C83.scad
python AutoKey.py --key 1,2,3,4,5 --profile profiles/AB-AB1.svg --definition definitions/AB-C83.scad

# Run all tests
python -m pytest tests/

# Run a single test class or test
python -m pytest tests/test_app.py::TestProfileIndex
python -m pytest tests/test_app.py::TestProfileDetail::test_combo_entry_enabled_in_key_mode

# Install UI dependencies
pip install -r requirements.txt   # Pillow only; svglib/reportlab are NOT needed
```

## Architecture

### Three-layer design

The project has an original CLI layer, a Tkinter desktop UI, and a Flask web app:

- **`autokey_core.py`** — pure generation logic (extracted from the original `AutoKey.py`). Importable by both UIs. Key functions: `generate_key()`, `write_profile_scad()`, `write_system_scad()`.
- **`AutoKey.py`** — thin CLI wrapper that delegates to `autokey_core.generate_key()`. Original interface preserved.
- **`autokey_ui.py`** — entry point for the Tkinter desktop app.
- **`ui/`** — all Tkinter UI code.
- **`web_app.py`** — Flask web server (pages + REST API).
- **`web/templates/`** — Jinja2 templates (extend `base.html`).
- **`web/static/`** — static assets (Bootstrap loaded from CDN).

### Two separate data concepts

A key model requires **two independent files** used together:
1. **Profile** (`profiles/<name>.scad` + `.svg` + `.dxf`) — the keyway shape. Parameters: `tol` (tolerance), `ph` (profile height). Optional: `khcx`, `khcz`, `thin_handle`, `match_handle`.
2. **System definition** (`definitions/<name>.scad`) — the lock's mechanical spec. Parameters: `kl` (key length), `aspace`, `pinspace`, `hcut`, `cutspace`, `cutangle`, `platspace`.

### Generation pipeline

`generate_key()` in `autokey_core.py`:
1. Reads profile `.scad` + system `.scad` content
2. Generates `branding/branding.dxf` via Inkscape + pstoedit
3. Copies pre-computed `profiles/<name>.dxf` → `profile.dxf` (skips Inkscape if DXF exists)
4. Composes `settings.scad` from all parts
5. Launches `openscad key.scad` as a non-blocking subprocess

`settings.scad` is auto-generated on each run — do not edit it manually.

### Tkinter UI navigation

The app uses a single `Tk` window with a frame-stack. All screens (`HomeScreen`, `ProfileDetail`, `AddProfile`, `AddSystem`) are instantiated once at startup and raised via `frame.tkraise()`. `App.show_screen(name, **kwargs)` calls `frame.refresh(**kwargs)` before raising. The `App` instance holds the single `ProfileIndex` and is passed to every screen at construction.

### Web app navigation

All pages extend `web/templates/base.html` (dark navbar with Bootstrap 5 + Bootstrap Icons). The `active_page` variable passed from each route highlights the current nav link.

| Route | Purpose |
|-------|---------|
| `/` | Generate key (profile grid → configure → download STL) |
| `/profiles` | Browse all profiles (thumbnail grid) |
| `/profile/<name>` | View + in-place edit a single profile |
| `/systems` | Browse all systems (parameter table) |
| `/system/<name>` | View + in-place edit a single system |
| `/add-profile` | Add a new profile |
| `/add-system` | Add a new system |

Edit pages use a JS view/edit toggle (no page reload). Save calls `PUT /api/profiles/<name>` or `PUT /api/systems/<name>`, which rewrites the `.scad` file and updates `profiles.json`.

### Data index

`profiles.json` is the shared data source for both the Tkinter UI and the web app. It is auto-generated by `ProfileIndex.rebuild_from_disk()` on first run by scanning `.scad` files. Subsequent runs load it directly. It is updated incrementally when adding or editing profiles/systems.

`ProfileIndex` (`ui/profile_index.py`) key methods: `load()`, `save()`, `add_profile()`, `add_system()`, `update_profile()`, `update_system()`, `set_profile_default_system()`.

### SVG thumbnails

`ui/svg_renderer.py` uses Inkscape to convert SVGs to PNG thumbnails cached in `%TEMP%/autokey3d_thumbs/`. Pillow loads the PNGs into `ImageTk.PhotoImage`. No Cairo dependency. The web app serves the raw `.svg` files directly — browsers render SVG natively, so no conversion is needed server-side.

### Testing

`tests/test_app.py` uses a single module-level `tk.Tk()` root (Tkinter only supports one per process). UI screens are tested by constructing them with a `_MockApp` stub, driving them via `StringVar.set()` / direct method calls, and asserting on widget state. `autokey_core.generate_key` and `tkinter.messagebox` are patched to avoid real subprocess calls.

## External tool dependencies

OpenSCAD, Inkscape, and pstoedit must be on `PATH`. They are invoked as subprocesses. Inkscape is used for branding SVG→EPS→DXF conversion and for SVG thumbnail generation (Tkinter UI only). The pre-computed `.dxf` files in `profiles/` allow key generation without Inkscape for the profile step.
