## Commands

All commands must bu runned in the venv.

```bash
# Launch the Tkinter UI
python autokey_ui.py

# CLI key generation (backward-compatible)
python AutoKey.py --bumpkey --profile profiles/AB-AB95.svg --definition definitions/AB-E20.scad
python AutoKey.py --blank   --profile profiles/AB-AB1.svg  --definition definitions/AB-C83.scad
python AutoKey.py --key 1,2,3,4,5 --profile profiles/AB-AB1.svg --definition definitions/AB-C83.scad

# Run all tests
python -m pytest tests/

# Run a single test class or test
python -m pytest tests/test_app.py::TestProfileIndex
python -m pytest tests/test_app.py::TestProfileDetail::test_combo_entry_enabled_in_key_mode

# Install UI dependencies
pip install -r requirements.txt   # Pillow only; svglib/reportlab are NOT needed
```

## Architecture

### Two-layer design

The project has an original CLI layer and a new UI layer added on top:

- **`autokey_core.py`** ŌĆö pure generation logic (extracted from the original `AutoKey.py`). Importable by the UI. Key functions: `generate_key()`, `write_profile_scad()`, `write_system_scad()`.
- **`AutoKey.py`** ŌĆö thin CLI wrapper that delegates to `autokey_core.generate_key()`. Original interface preserved.
- **`autokey_ui.py`** ŌĆö entry point for the Tkinter app.
- **`ui/`** ŌĆö all UI code.

### Two separate data concepts

A key model requires **two independent files** used together:
1. **Profile** (`profiles/<name>.scad` + `.svg` + `.dxf`) ŌĆö the keyway shape. Parameters: `tol` (tolerance), `ph` (profile height). Optional: `khcx`, `khcz`, `thin_handle`, `match_handle`.
2. **System definition** (`definitions/<name>.scad`) ŌĆö the lock's mechanical spec. Parameters: `kl` (key length), `aspace`, `pinspace`, `hcut`, `cutspace`, `cutangle`, `platspace`.

### Generation pipeline

`generate_key()` in `autokey_core.py`:
1. Reads profile `.scad` + system `.scad` content
2. Generates `branding/branding.dxf` via Inkscape + pstoedit
3. Copies pre-computed `profiles/<name>.dxf` ŌåÆ `profile.dxf` (skips Inkscape if DXF exists)
4. Composes `settings.scad` from all parts
5. Launches `openscad key.scad` as a non-blocking subprocess

`settings.scad` is auto-generated on each run ŌĆö do not edit it manually.

### UI navigation

The app uses a single `Tk` window with a frame-stack. All screens (`HomeScreen`, `ProfileDetail`, `AddProfile`, `AddSystem`) are instantiated once at startup and raised via `frame.tkraise()`. `App.show_screen(name, **kwargs)` calls `frame.refresh(**kwargs)` before raising. The `App` instance holds the single `ProfileIndex` and is passed to every screen at construction.

### Data index

`profiles.json` is the UI's data source. It is auto-generated by `ProfileIndex.rebuild_from_disk()` on first run by scanning `.scad` files. Subsequent runs load it directly. It is updated incrementally when adding profiles/systems via the UI forms.

### SVG thumbnails

`ui/svg_renderer.py` uses Inkscape (already required) to convert SVGs to PNG thumbnails cached in `%TEMP%/autokey3d_thumbs/`. Pillow loads the PNGs into `ImageTk.PhotoImage`. No Cairo dependency.

### Testing

`tests/test_app.py` uses a single module-level `tk.Tk()` root (Tkinter only supports one per process). UI screens are tested by constructing them with a `_MockApp` stub, driving them via `StringVar.set()` / direct method calls, and asserting on widget state. `autokey_core.generate_key` and `tkinter.messagebox` are patched to avoid real subprocess calls.

## External tool dependencies

OpenSCAD, Inkscape, and pstoedit must be on `PATH`. They are invoked as subprocesses. Inkscape is used for branding SVGŌåÆEPSŌåÆDXF conversion and for SVG thumbnail generation. The pre-computed `.dxf` files in `profiles/` allow key generation without Inkscape for the profile step.
